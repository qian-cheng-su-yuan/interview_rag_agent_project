# 项目介绍：企业知识库 AI Agent 问答系统

## 一、项目背景

在企业内部，制度文件、报销流程、入职说明、考勤规范等资料通常分散在多个文档中。员工遇到问题时，往往需要人工查找或询问行政、人事、财务等部门，效率较低。传统关键词搜索只能返回文档位置，不能直接给出整理后的回答。

本项目基于 RAG（Retrieval-Augmented Generation，检索增强生成）思想，设计并实现一个企业知识库 AI Agent 问答系统。系统支持上传企业内部资料，将资料切分为文本片段并建立本地检索索引。用户输入问题后，系统先检索相关片段，再将片段和问题组合成 Prompt 调用大模型生成回答，同时展示参考片段，降低模型胡编乱造的风险。

## 二、项目目标

本项目主要实现以下目标：

1. 支持企业文档上传和解析，包括 TXT、MD、PDF。
2. 支持文档清洗、分段切片和本地向量检索。
3. 支持基于知识库内容的问答生成。
4. 支持参考片段展示，方便用户判断答案来源。
5. 支持文档总结、关键词提取和问答日志记录，体现 Agent 工具能力。
6. 采用企业中常见的分层项目结构，便于扩展和维护。

## 三、核心流程

```text
用户上传文档
   ↓
文档解析与文本清洗
   ↓
文本分段切片
   ↓
构建本地向量索引
   ↓
用户输入问题
   ↓
相似度检索相关片段
   ↓
拼接 Prompt
   ↓
调用大模型生成回答
   ↓
返回答案、参考片段、耗时和日志
```

## 四、技术设计

### 1. 后端设计

后端使用 FastAPI 实现，主要负责文档上传、知识库重建、问答接口、工具接口和日志接口。项目按企业项目习惯拆分为 API 层、Schema 层、Service 层、Core 层和 Utils 层。

### 2. 前端设计

前端使用 Streamlit 搭建本地演示页面。页面包含文档上传、问题输入、答案展示、参考片段、文档总结、关键词提取和最近问答日志，适合面试现场演示。

### 3. 检索设计

默认使用 scikit-learn 的 TF-IDF 向量化方法实现轻量级本地检索。这样做的好处是部署简单、依赖少、启动快。生产环境中可以将 `vector_store.py` 替换为 Chroma、FAISS、Milvus 或 Elasticsearch。

### 4. 大模型设计

系统通过 OpenAI-compatible Chat Completions 接口调用大模型。只要服务方兼容该协议，就可以通过 `.env` 修改 `LLM_BASE_URL`、`LLM_MODEL_NAME` 和 `LLM_API_KEY` 进行切换。

## 五、项目优势

- **可运行**：不是只有代码片段，而是完整项目文件。
- **可演示**：本地启动后可上传文档并提问。
- **可讲解**：RAG 主链路清晰，方便面试时说明自己的实现思路。
- **可扩展**：后续可替换向量库、增加登录权限、接入数据库和部署到服务器。
- **有兜底**：没有配置 API Key 时，也可以基于检索片段给出本地演示回答。

## 六、面试讲解话术

可以这样介绍本项目：

> 我做的是一个企业知识库 AI Agent 问答系统。系统支持上传企业内部资料，然后自动完成文档解析、文本切分、向量检索和大模型回答。用户提问时，系统不是直接让模型自由回答，而是先从知识库中检索相关内容，再把检索结果拼到 Prompt 里，让模型基于资料回答，并展示参考片段。这样可以降低幻觉，也方便用户追溯答案来源。项目结构上，我采用了 FastAPI 后端加 Streamlit 前端的方式，代码按 API、Schema、Service、Core 分层，比较接近企业常见的 Python 后端项目结构。

## 七、后续优化方向

1. 将本地 TF-IDF 检索替换为 Chroma 或 FAISS。
2. 增加多轮对话记忆，支持结合历史上下文追问。
3. 增加用户权限，限制不同用户访问不同知识库。
4. 增加文档增量更新，避免每次上传后全量重建。
5. 增加 Dockerfile 和服务器部署脚本。
6. 增加评测集，用于测试知识库问答准确率。
